EaseMotion CSS Bug Fix

Modal Component — Token Fix

components/modals.css referenced two undefined CSS variables — --ease-color-text-muted and --ease-shadow-2xl — which silently fell back to hardcoded light-mode values and never adapted to dark mode. This demo shows the modal rendered in a forced light scope and a forced dark scope, side by side, using the fixed modals.css.

The Bug

Before (broken)
.ease-modal-body {
  color: var(--ease-color-text-muted, #4b5563);
}
.ease-modal {
  box-shadow: var(--ease-shadow-2xl, 0 25px 50px -12px rgba(0,0,0,0.25));
}

Both custom properties are never defined in core/variables.css — every var() call falls through to the hardcoded fallback, in both light and dark mode. Modal body text stays #4b5563 forever; the shadow never gets the dark-mode-tuned value.

After (fixed)
.ease-modal-body {
  color: var(--ease-color-muted, #4b5563);
}
.ease-modal {
  box-shadow: var(--ease-shadow-xl, 0 25px 50px -12px rgba(0,0,0,0.25));
}

--ease-color-muted and --ease-shadow-xl are both real tokens with light AND dark mode values defined in core/variables.css — the modal now correctly adapts.
