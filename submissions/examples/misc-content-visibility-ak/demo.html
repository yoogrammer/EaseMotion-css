Content Visibility & Animations

content-visibility: auto on animated elements breaks scroll-based animations by skipping off-screen rendering.

Broken: content-visibility: auto

Animation is applied but the initial state is never painted off-screen.

⚠

Card with content-visibility

This card has content-visibility: auto. Its animation may not start correctly if loaded off-screen.

.card {
  content-visibility: auto;
  animation: fadeSlide 1s;
}
Animation may not start

Fixed: No content-visibility

Without content-visibility, the animation initializes correctly.

✔

Card without content-visibility

No content-visibility. The animation renders properly regardless of scroll position.

.card {
  /* no content-visibility */
  animation: fadeSlide 1s;
}
Animation works reliably

content-visibility: auto

With content-visibility: auto, the browser skips painting this element until it's near the viewport, breaking the scroll-linked animation timeline.

No content-visibility

Without content-visibility, the scroll-driven animation binds correctly and tracks progress smoothly.

When to avoid content-visibility on animated elements

  • Scroll-driven animations (view-timeline, scroll-timeline): The browser needs the initial painted frame to bind the timeline. content-visibility: auto skips this.
  • CSS animations on page load: If the element is below the fold, the animation may appear to "jump" when scrolled into view.
  • Workaround: Use contain-intrinsic-size to reserve space, or simply omit content-visibility from animated elements.
