Bug Fix: --ease-color-text-dark undefined token

In dark mode, .ease-card-glass references var(--ease-color-text-dark, #f8fafc) but the token is never defined in variables.css, so the browser always silently uses the hardcoded fallback #f8fafc.

💡 Toggle your OS dark mode to see both states. The page simulates both buggy and fixed cards side-by-side.

❌ Before (bug)
Hardcoded fallback

Glass Card (Buggy)

This card's dark-mode text color is set via var(--ease-color-text-dark, #f8fafc). Because --ease-color-text-dark is not defined in variables.css, the browser falls back to the hardcoded value #f8fafc — bypassing the design token system.

Computed text color: — 
/* components/cards.css — current (buggy) */
@media (prefers-color-scheme: dark) {
  .ease-card-glass {
    color: var(--ease-color-text-dark, #f8fafc);
    /* ↑ token never defined → always uses #f8fafc */
  }
}
✅ After (fix)
Token-driven

Glass Card (Fixed)

With --ease-color-text-dark: #f8fafc properly declared inside the dark-mode block of variables.css, the token resolves correctly. Consumers can now override the color by redefining --ease-color-text-dark in their own CSS without touching any component file.

Computed text color: — 
/* Proposed fix — variables.css (dark-mode block) */
@media (prefers-color-scheme: dark) {
  :root {
    --ease-color-text-dark: #f8fafc; /* ← add this */
  }
}

/* components/cards.css — after fix */
@media (prefers-color-scheme: dark) {
  .ease-card-glass {
    color: var(--ease-color-text-dark); /* no fallback needed */
  }
}

Why the token matters — easy customisation

Once the token exists, consumers can theme it without touching any framework file: 

/* In your own stylesheet */
:root {
  --ease-color-text-dark: #cdd6f4; /* Catppuccin Mocha text */
}
/* Every dark-mode glass card in the project updates automatically. */
