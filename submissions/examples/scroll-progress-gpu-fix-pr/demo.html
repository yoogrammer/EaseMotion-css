🐛 Scroll-Progress GPU Memory Leak Fix

Fixes an issue where will-change: transform was unconditionally applied, permanently wasting GPU memory on low-end devices.

What changed?

The progress bar is fixed at the top of the page. Because it is permanently in the DOM, applying will-change to it globally forces the browser to keep a dedicated hardware compositing layer open forever. The fix applies the hint only when the animation is actually supported and active.

❌ Before (Line 46)
.ease-scroll-progress {
  /* Always active GPU layer! */
  will-change: transform;
  
  animation: ease-kf auto linear;
  animation-timeline: scroll();
}
✅ After (Fixed)
.ease-scroll-progress {
  /* Default: let browser decide */
  will-change: auto;
}

@supports (animation-timeline: scroll()) {
  .ease-scroll-progress {
    /* Only apply when animating */
    will-change: transform;
  }
}
How to test:
1. Scroll down the page — you will see the purple scroll progress bar at the very top of the screen filling up from left to right.
2. The visual behavior is exactly the same as before.
3. However, if you open Chrome DevTools → Layers panel, or inspect the element in environments without animation-timeline support, or if you have "Reduce Motion" enabled in your OS, the browser is no longer forced to allocate unnecessary GPU memory.
4. This provides a significant battery and performance boost for mobile devices.
