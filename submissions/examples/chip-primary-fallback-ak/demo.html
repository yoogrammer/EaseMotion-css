Chip: Primary Color Fallback

Wrong fallback #6366f1 vs correct fallback #6c63ff in chip component.

#6366f1 Wrong (Tailwind Indigo-500)
#6c63ff Correct (EaseMotion Primary)

Before: Wrong Fallback

Uses #6366f1 as the fallback — a Tailwind color, not EaseMotion's primary.

Primary Active Selected 
.ease-chip-primary {
  background: var(
    --ease-color-primary,
    #6366f1     /* WRONG */
  );
  color: #fff;
}

#6366f1 is Indigo-500 from Tailwind, not EaseMotion's brand primary. When the CSS custom property is undefined, the chip renders in the wrong color, breaking brand consistency.

After: Correct Fallback

Uses #6c63ff — the actual EaseMotion primary color.

Primary Active Selected 
.ease-chip-primary {
  background: var(
    --ease-color-primary,
    #6c63ff     /* CORRECT */
  );
  color: #fff;
}

#6c63ff is EaseMotion's documented primary color. When the custom property is undefined, the chip renders correctly, maintaining brand consistency.

Why this matters

The fallback value in var() is the safety net when the CSS custom property is not defined. Using a wrong fallback color:

  • Breaks brand consistency when tokens are missing
  • Makes the component look like it belongs to a different design system
  • Confuses developers who expect EaseMotion's actual primary color
